# systemd-service-builder

Generate **production-ready, security-hardened `systemd` service unit files**
from the command line or a small JSON config.

Most hand-written `.service` files miss the dozen-or-so directives that
`systemd-analyze security` calls out as low-hanging fruit
(`NoNewPrivileges`, `ProtectSystem=strict`, `RestrictAddressFamilies`,
`SystemCallFilter`, dropped capabilities, etc.). This tool wires them in by
default so your daemon ships with a sane sandbox from day one — and lets you
override anything that doesn't fit your service.

- **Zero dependencies** — single Python 3.8+ file, stdlib only.
- **Hardened by default** — opinionated, override-friendly directive set.
- **Two ways to drive it** — flags (`--name`, `--exec`, ...) or `--config file.json`.
- **Validated** — bad service names, bogus `Restart=` values, malformed
  `MemoryMax`, etc. fail fast with a clear error.
- **Helpful warnings** — flags relative paths, `User=root`, missing
  `ReadWritePaths` under hardened mode.

## Install

```bash
# Direct script (recommended)
curl -O https://raw.githubusercontent.com/intruderfr/systemd-service-builder/main/systemd_service_builder.py
chmod +x systemd_service_builder.py
sudo mv systemd_service_builder.py /usr/local/bin/systemd-service-builder
```

Or via `pip` from a clone:

```bash
git clone https://github.com/intruderfr/systemd-service-builder.git
cd systemd-service-builder
pip install .
```

## Quick start (flags)

```bash
systemd-service-builder \
  --name node-api \
  --description "Internal Node.js API server" \
  --exec "/usr/bin/node /opt/node-api/server.js" \
  --user nodeapi --group nodeapi \
  --workdir /opt/node-api \
  --env-file /etc/node-api/env \
  --memory-max 1G \
  --cpu-quota 200% \
  --limit-nofile 65535 \
  --rw-path /var/log/node-api \
  --rw-path /var/lib/node-api \
  --cap CAP_NET_BIND_SERVICE \
  -o node-api.service
```

## Quick start (config file)

`node-api.json`:

```json
{
  "name": "node-api",
  "description": "Internal Node.js API server",
  "exec_start": "/usr/bin/node /opt/node-api/server.js",
  "user": "nodeapi",
  "group": "nodeapi",
  "working_directory": "/opt/node-api",
  "environment_file": "/etc/node-api/env",
  "memory_max": "1G",
  "cpu_quota": "200%",
  "limit_nofile": 65535,
  "read_write_paths": ["/var/log/node-api", "/var/lib/node-api"],
  "capability_bounding_set": "CAP_NET_BIND_SERVICE"
}
```

```bash
systemd-service-builder --config node-api.json -o node-api.service
```

Both invocations produce the same unit file — see
[`examples/node-api.service`](examples/node-api.service).

A typical output looks like this (truncated):

```ini

Group=nodeapi

EnvironmentFile=/etc/node-api/env
ExecStart=/usr/bin/node /opt/node-api/server.js
Restart=on-failure
RestartSec=5
MemoryMax=1G
CPUQuota=200%
LimitNOFILE=65535

# --- hardening (generated by systemd-service-builder) ---
NoNewPrivileges=true
PrivateTmp=true
PrivateDevices=true
ProtectSystem=strict
ProtectHome=true
ProtectKernelTunables=true
ProtectKernelModules=true
ProtectKernelLogs=true
ProtectControlGroups=true
ProtectClock=true
ProtectHostname=true
ProtectProc=invisible
ProcSubset=pid
RestrictNamespaces=true
RestrictRealtime=true
RestrictSUIDSGID=true
```

## Hardening defaults

| Directive | Default | Why |
|---|---|---|
| `NoNewPrivileges` | `true` | Block setuid escalation. |
| `PrivateTmp` | `true` | Service gets its own `/tmp`. |
| `PrivateDevices` | `true` | No raw device access. |
| `ProtectSystem` | `strict` | Whole filesystem read-only by default — opt in via `ReadWritePaths`. |
| `ProtectHome` | `true` | `/home`, `/root`, `/run/user` invisible. |
| `ProtectKernel*` | `true` | Kernel tunables/modules/logs locked down. |
| `ProtectControlGroups` | `true` | cgroup tree read-only. |
| `ProtectProc` | `invisible` | Other processes invisible in `/proc`. |
| `RestrictNamespaces` | `true` | No new namespaces. |
| `RestrictAddressFamilies` | `AF_INET AF_INET6 AF_UNIX` | No exotic sockets. |
| `SystemCallFilter` | `@system-service` | Strict syscall allowlist. |
| `CapabilityBoundingSet` | _(empty)_ | Drop ALL caps. Override with `--cap CAP_NET_BIND_SERVICE` etc. |
| `MemoryDenyWriteExecute` | `true` | Block W^X bypass. |
| `LockPersonality` | `true` | No `personality(2)` shenanigans. |
| `UMask` | `0077` | Created files default to owner-only. |

Disable the whole block with `--no-harden` if you're generating a unit for a
service that genuinely needs it (e.g. something that loads kernel modules).

## CLI reference

```
usage: systemd-service-builder [-h] [--config CONFIG] [--name NAME]
                               [--description DESCRIPTION] [--exec EXEC_START]
                               [--user USER] [--group GROUP]
                               [--workdir WORKING_DIRECTORY] [--env ENV]
                               [--env-file ENVIRONMENT_FILE] [--type TYPE]
                               [--restart RESTART] [--restart-sec RESTART_SEC]
                               [--memory-max MEMORY_MAX] [--cpu-quota CPU_QUOTA]
                               [--tasks-max TASKS_MAX]
                               [--limit-nofile LIMIT_NOFILE] [--rw-path RW_PATH]
                               [--ro-path RO_PATH] [--cap CAP] [--no-harden]
                               [--output OUTPUT] [--print-spec] [--quiet]
```

`--config` and direct flags can be combined: the config is loaded first and
flags override individual fields.

## Installing the generated unit

```bash
sudo cp node-api.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now node-api.service
sudo systemctl status node-api.service
sudo journalctl -u node-api.service -f
```

After it's running, check the security score:

```bash
systemd-analyze security node-api.service
```

A correctly-hardened unit from this tool typically scores **2–3 (OK)** —
the remaining points usually require `User=` and `DynamicUser=` decisions only
you can make.

## Programmatic use

The script is also importable as a module:

```python
from systemd_service_builder import ServiceSpec, render, validate

spec = ServiceSpec(
    name="my-app",
    description="My demo daemon",
    exec_start="/usr/bin/my-app --serve",
    user="myapp",
    read_write_paths=["/var/lib/my-app"],
)
warnings = validate(spec)
unit_text = render(spec)
```
